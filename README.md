# 吊装作业 3D 复盘标注系统

基于 React + Three.js 的吊装过程三维可视化复盘与风险标注工具。

## 功能特性

- **3D 场景回放**：基于 Three.js + react-three-fiber 的吊装轨迹回放
- **风险批注**：在时间轴任意位置添加、编辑、删除风险批注（安全/警告/危险三级）
- **风险筛选**：按风险等级多选筛选，支持一键忽略/显示已忽略风险
- **复盘快照**：将任意时刻的镜头、时间轴位置、筛选条件、可见批注固化为不可变快照
  - 快照命名保存、覆盖更新、撤销最近一次改动
  - 作业间隔离，切换作业快照不串档
  - 刷新/重开后快照仍可用（localStorage 持久化）
- **会话包工作台**：交付包制作、版本管理和回放入口一站式管理
  - 打包当前时间轴、相机视角、筛选条件、批注、模板来源、统计摘要和 JSON/CSV 导出产物为一份会话包
  - 列表直接显示版本号、发布时间、是否过期及过期原因
  - 支持覆盖当前版本、另存新版本（版本号自增/自定义）、撤回最近一次发布
  - 导入会话包自动恢复场景、批注、筛选、统计和相机视角；版本冲突时三选一处理（改名/覆盖/取消）
  - 批注或筛选条件变动后，旧包自动标记过期，无法继续导出
  - 所有操作（成功/失败/冲突/过期）记入本地操作日志，刷新或重开应用后仍可查询
- **会话轨迹归档**：会话包从场景快照升级为可复盘全过程的完整档案
  - 发布、覆盖、另存、撤回、导入冲突、取消导入、导入失败、恢复回放等所有动作均生成结构化操作记录
  - 每条操作记录附带完整上下文：时间轴位置、相机视角、筛选条件、统计摘要、模板来源、导出产物
  - 冲突处理分三步独立日志：弹窗出现（import_conflict_detected）→ 用户选择（import_conflict_cancel/rename/overwrite）→ 最终结果（import）
  - 操作记录随会话包一起封包，导入或回放时包内历史日志自动同步回本地（按 ID 去重）
  - 刷新页面或重开应用后，所有操作日志按时间顺序可追溯，操作链条完整可审计
  - 包体内置整体校验和（checksum），防止内容篡改
- **报告导出**：
  - JSON：完整作业元数据 + 轨迹 + 批注 + 快照信息
  - CSV：轨迹表格 + 批注表格，可直接用 Excel 打开
  - 所有导出均基于快照，不受导出时筛选条件变动影响
- **相机预置**：俯视/侧视/正视/跟随吊钩等常用视角

## 启动项目

### 环境要求
- Node.js >= 18
- npm >= 9

### 安装依赖
```bash
npm install
```

### 开发模式
```bash
npm run dev
```
默认启动在 http://localhost:5173/

### 生产构建
```bash
npm run build
```

### 预览生产构建
```bash
npm run preview
```
默认启动在 http://localhost:4173/

### 运行测试
```bash
npm test
```

### 类型检查
```bash
npm run check
```

## 完整操作流程（可复现）

### 1. 加载数据

首屏有两个面板入口，均为固定悬浮：

**左上角「数据导入」面板**：点击「数据导入 ▼」按钮展开面板
- 面板上半区是灰色上传区，文案为 **「拖放 JSON 文件或点击选择」**
- 面板下半区有 **「加载样例数据」** 按钮
- 导入失败时错误列表面板内显示，首行标题：**「预检未通过，当前作业保持不变」**

方式 A：加载内置样例
- 展开左上角「数据导入 ▼」面板
- 点击面板内 **「加载样例数据」** 按钮
- 成功后面板底部出现绿色 **「数据加载成功」**，下方列有作业名称、日期、起重机编号
- 样例数据来自 `src/data/sampleJob.ts`，不含预置批注（右侧批注面板显示「暂无风险批注」）

方式 B：导入自定义 JSON
- 展开左上角「数据导入 ▼」面板
- 点击面板内灰色上传区（文案「拖放 JSON 文件或点击选择」），或直接把 `.json` 文件拖入该区域
- 校验通过时状态同花式 A
- 校验不通过时：面板内显示红色标题「预检未通过，当前作业保持不变」，下方逐条列出红色错误信息（不会写进浏览器 console）；**当前作业、已有批注、忽略列表、快照选中状态与导出能力、作业名称、日期、起重机编号全部原样保留，不会被清空**；此时「数据加载成功」绿色提示会自动隐藏，仅保留作业元信息

### 2. 调整筛选条件

**右上角「风险批注」面板**：点击「风险批注」按钮展开右侧面板，顶部是筛选区：
- **风险等级**：**「安全」「警告」「危险」** 三个可独立切换的按钮，激活时显示对应颜色徽章，未激活时灰色描边
- **显示忽略开关**：位于筛选区最右，图标 ToggleLeft/ToggleRight，`sm` 屏以上右侧附带文字「显示忽略」/「隐藏忽略」
- 无批注时列表区显示 **「暂无风险批注」**

### 3. 生成导出快照

点击**顶部中央**的「导出报告」按钮打开导出面板：

1. 面板标题 **「复盘快照」**，右上角 **「新建」** 按钮
2. 未建快照时面板提示：**「暂无快照，点击"新建"创建导出快照」**
3. 点击「新建」，弹窗标题 **「创建快照」**，输入框 placeholder **「输入快照名称」**，默认已填充日期时间
4. 点击弹窗右下角 **「保存」** 完成创建
5. 快照创建成功后，面板显示：
   - 绿勾 + **「当前快照：{名称}」**
   - 三格统计：**「风险」（红字）/「批注」（青字）/「已忽略」（灰字）**
   - 筛选描述行：**「筛选：含已忽略 \| 安全/警告/危险」**（按实际状态）
   - 三个操作按钮：**「覆盖更新」「撤销」「预览」**
   - 下方区块标题 **「历史快照」**，列出同名作业的快照（可点击切换，带删除图标）

**快照固化的内容**（创建后不再变化）：
- 时间轴当前播放位置
- 3D 镜头的位置和朝向
- 是否显示已忽略批注
- 风险等级筛选开关状态
- 当前可见的所有批注（带批注文本、等级、时间戳）
- 作业元信息、完整轨迹、起重机配置、限制区域

### 4. 预览快照内容

点击「预览」按钮，面板底部展开 **「快照数据预览」** 区：
- 最多展示前 5 条批注，含文本、等级圆点、时间戳、忽略状态
- 超过 5 条显示：**「还有 N 条批注...」**

### 5. 导出报告

快照创建后，面板底部两个导出按钮从禁用变可点：

**「导出 JSON」**（附小字：完整轨迹数据 + 快照风险批注）
- 文件名格式：`吊装复盘_{作业名}_{快照名}_{日期}.json`
- 内容结构：
  ```json
  {
    "meta": { "name": "...", "date": "...", "craneId": "...", ... },
    "crane": { ... },
    "restrictedZones": [ ... ],
    "trajectory": [ ... ],
    "annotations": [ ... ],
    "riskStats": { "total": N, "danger": N, "warning": N, "safe": N, "ignored": N, "visible": N, "exported": N },
    "exportedAt": "...",
    "snapshotInfo": {
      "id": "...", "name": "...", "createdAt": "...",
      "updatedAt": "...", "currentTime": N, "camera": { ... }
    },
    "exportOptions": {
      "includeIgnored": true/false,
      "riskLevelFilter": { "safe": t/f, "warning": t/f, "danger": t/f },
      "ignoredCount": N, "visibleCount": N, "exportedCount": N
    }
  }
  ```

**「导出 CSV」**（附小字：轨迹表格 + 批注表格，可用 Excel 打开）
- 文件名格式：`吊装复盘_{作业名}_{快照名}_{日期}.csv`
- 内容按区块分为（区块标题使用 `=== 标题 ===`）：
  1. 快照信息（名称、ID、创建/更新时间、时间轴位置、相机位置/目标）
  2. 筛选条件（包含已忽略、各等级开关、忽略列表）
  3. 轨迹数据表格
  4. 风险批注表格
  5. 风险统计汇总

面板最底部提示行：
- 已选快照：**「导出内容基于快照数据，与创建快照时的筛选条件一致」**
- 未选快照：**「请先创建快照后再导出」**

### 6. 核对导出结果

**JSON 核对**：
1. 用任意文本编辑器打开导出的 `.json` 文件
2. 核对 `riskStats.exported` 与面板「批注」格数字一致
3. 核对 `exportOptions.riskLevelFilter` 与面板「筛选：」行描述一致
4. 数 `annotations` 数组长度应等于 `riskStats.exported`

**CSV 核对**：
1. 用 Excel 或 WPS 打开 `.csv` 文件（UTF-8 编码）
2. 滚动到 `=== 风险统计 ===` 区块，对比「批注总数」「危险」「警告」「安全」「已忽略」「可见」「导出」各行数值
3. `=== 风险批注 ===` 区块下以 `ann-` 开头的行数应等于统计中的「导出」数量

### 7. 覆盖更新与撤销

**「覆盖更新」**：
- 当前筛选条件或批注内容改动后，点击即按当前状态刷新快照
- 若筛选条件已变，弹窗提示：**「筛选条件已变更，覆盖更新将使用当前筛选条件，是否继续？」**
- 每次覆盖更新自动推入历史栈，可撤销

**「撤销」**：
- 回退到上一次覆盖更新前的版本
- 可连续撤销，直到历史栈为空
- 无可撤销时按钮自动变灰禁用

筛选条件与快照不一致时，面板顶部黄色警告：
**「筛选条件已变更，导出将使用快照中的数据。建议更新快照后再导出。」**
此时点导出会再弹窗：
**「筛选条件已变更，导出将使用快照中的筛选条件。\n\n建议重新生成快照后再导出。\n\n是否继续导出？」**

### 8. 刷新 / 重开验证

1. 创建好快照后，按 F5 刷新页面
2. 页面重新加载后，点击顶部中央「导出报告」
3. 「历史快照」区仍列出之前创建的快照（持久化来自 localStorage）
4. 点击选中后，预览、JSON 导出、CSV 导出均可继续操作

### 9. 切换作业隔离验证

1. 为作业 A 创建若干快照并选中其中一个
2. 展开左上角「数据导入 ▼」面板，重新加载样例或导入另一份不同作业
3. 点击「导出报告」：
   - 当前选中状态已清空（`currentSnapshotId = null`）
   - 「历史快照」区只显示当前作业的快照（作业 A 的快照不出现在列表中，`snapshots` 以 `jobId` 分桶隔离）
4. 再导回作业 A：原来的快照全部恢复可见

## 会话包工作台操作流程（可复现）

点击**顶部中央**的「会话包工作台」按钮打开工作台面板，面板内有两个 Tab：**会话包列表** 和 **操作日志**。

入口位置：顶部中央按钮组，「导出报告」右侧，按钮文案 **「会话包工作台」**。

---

### 链路 1：发布会话包 + 覆盖 + 另存 + 撤回

#### 1.1 首次发布

1. 加载样例数据后（见前文「加载样例数据」），在右侧批注面板添加 2~3 条批注
2. 打开「会话包工作台」面板，面板内显示：**「暂无会话包」**，下方提示 **「点击右上角『发布』创建首个会话包」**
3. 点击面板右上角 **「发布」** 按钮，弹窗标题 **「发布会话包」**
4. 默认填充：
   - 会话包名称：`{作业名}-会话包`
   - 版本号：`1.0.0`（首次创建）
5. 点击弹窗右下角 **「发布」** 按钮
6. 成功提示：顶部中央弹出绿色 Toast **「发布会话包成功」**
7. 面板内容变化：
   - 顶部出现当前包卡片，绿色背景，显示包名、版本徽章 `v1.0.0`、发布时间
   - 下方四格统计：批注数、危险数、时间轴位置（ms）、模板来源数
   - 下方列表出现该包条目，左侧绿色圆点（有效）、右侧 3 个操作图标：预览、回放、导出
   - 列表条目包名右侧出现 ✨ 图标（表示最近一次发布）

#### 1.2 覆盖更新当前版本

1. 在右侧批注面板新增或修改一条批注内容
2. 观察当前包卡片：顶部变为**琥珀色背景**，标题显示 **「已过期 · 批注数据已变更」**，列表条目左侧圆点也从绿色变为琥珀色
3. 在包卡片下方点击 **「覆盖」** 按钮（6 个操作按钮第 1 个）
4. 弹出确认框：标题 **「覆盖更新」**，内容提示 **「当前时间轴、相机视角、筛选条件和批注将写入此包，旧内容将被替换」**
5. 点击 **「确认覆盖」**
6. 成功提示：绿色 Toast **「覆盖更新成功」**
7. 包卡片恢复绿色背景，列表条目恢复绿色圆点，包状态重新变为「有效」

#### 1.3 另存为新版本

1. 再添加一条批注，当前包再次变为琥珀色（已过期）
2. 点击包卡片 **「另存」** 按钮（第 2 个按钮）
3. 弹窗标题 **「另存新版本」**，显示原版本名称和版本号，下方新版本号默认填入 **「1.0.1」**（版本号自增）
4. 可点击 **「自增」** 按钮继续自增，或手动修改版本号
5. 点击 **「另存」** 按钮
6. 成功提示：绿色 Toast **「另存版本 1.0.1 成功」**
7. 列表现在有 2 个包：v1.0.1 绿色有效 + ✨（最近发布），v1.0.0 琥珀色已过期

#### 1.4 撤回最近一次发布

1. 确认 v1.0.1 包是最近发布的（带 ✨ 图标）
2. 点击包卡片 **「撤回」** 按钮（第 3 个按钮）
3. 确认弹窗：**「确定要撤销最近一次发布吗？撤销后该包将标记为已过期。」** → 确定
4. 成功提示：绿色 Toast **「撤销发布成功」**
5. v1.0.1 变为琥珀色，状态 **「已过期 · 已撤销发布」**，✨ 图标消失
6. 此时「撤回」按钮变灰禁用（因为已无最近发布）

---

### 链路 2：导入冲突处理

#### 2.1 导出一个会话包作为导入源

1. 在列表中选中 v1.0.0 或 v1.0.1 包（状态为有效时）
2. 点击包条目右侧的 **「导出」** 图标 ↓，或包卡片的「导出」按钮
3. 浏览器下载文件：`会话包_{包名}_v{版本号}_{日期}.json`
4. 记下此文件的版本号（如 v1.0.0），稍后用于制造冲突

#### 2.2 制造版本冲突并处理

1. 在工作台面板中，用相同版本号再发布一个包：点击「发布」，版本号手动改为与导出文件相同（如 1.0.0）
   - 若系统提示版本号已存在，可先改为 1.0.2 发布后再删除或覆盖，确保本地已有一个与导入文件版本相同的包
2. 点击面板右上角 **「导入」** 按钮 ↑，选择刚才导出的 `.json` 文件
3. 如果版本号与本地已有包冲突，弹出 **「版本冲突」** 对话框，包含：
   - 左右对比卡片：左侧「现有版本」（灰色）、右侧「待导入版本」（琥珀色边框），分别显示包名、版本号、发布时间
   - 三个单选项：
     - **改名后导入**（默认选中）：下方输入框自动填入自增后的新版本号，如 `1.0.3`
     - **覆盖现有版本**：提示现有包内容将被替换
     - **取消导入**：跳过此包

4. **场景 A：改名导入**（默认）
   - 保持「改名后导入」选中，确认新版本号（如 1.0.3），点击「确认」
   - 成功提示：绿色 Toast **「重命名为 v1.0.3 导入成功」**
   - 列表中出现新包，版本号为修改后的 v1.0.3，状态有效

5. **场景 B：覆盖导入**
   - 重新执行步骤 2（再次导入同一文件）
   - 在冲突对话框中选择 **「覆盖现有版本」**，点击「确认」
   - 成功提示：绿色 Toast **「覆盖导入成功」**
   - 本地对应版本包的内容被替换

6. **场景 C：取消导入**
   - 第三次执行导入
   - 在冲突对话框中选择 **「取消导入」**，点击「确认」
   - 对话框关闭，本地列表无变化

---

### 链路 3：重启后回放 + 日志持久化

#### 3.1 回放恢复（验证包内场景恢复）

1. 确保列表中有至少 1 个有效包
2. 调整当前状态：拖动时间轴到任意位置、改变相机视角（3D 场景中鼠标拖拽旋转）、关闭某个风险等级筛选
3. 点击包条目右侧的 **「回放」** 图标 ▶，或包卡片的「回放」按钮
4. 确认弹窗：**「确定要恢复到会话包『{包名} v{版本}』吗？\n当前作业、批注、筛选、相机视角将被替换。」** → 确定
5. 成功提示：绿色 Toast **「回放恢复成功」**
6. 验证恢复效果：
   - 时间轴位置：跳转回包创建时的 `{时间轴}ms`
   - 相机视角：3D 场景恢复到包固化时的位置

## 会话轨迹归档

会话轨迹归档将会话包从单一的场景快照文件，升级为**可复盘全过程的完整档案**。每一次发布、覆盖、另存、撤回、导入、冲突处理、恢复回放等关键动作，都会生成**结构化操作记录**，连同操作时的完整上下文（时间轴、相机、筛选、统计、模板、导出产物）一起封入包内。导入或回放时，包内的历史操作记录会自动同步回本地日志，确保操作链条完整可追溯。

## 会话审计恢复

会话审计恢复模块在「会话轨迹归档」基础上，进一步确保**操作历史在清空日志、换机器、改配置或重开应用后仍可完整恢复**。

### 核心机制

1. **导出封包**：导出会话包时，完整操作历史（`operationLogs`）随包一起序列化导出
2. **导入恢复**：导入会话包时，先用 `mergeAndSortLogs` 按顺序将包内历史记录恢复到本地（按 ID 去重），再追加本次导入日志
3. **回放恢复**：回放会话包时，先将包内历史记录恢复到本地，再写入 `audit_restore` 类型日志（含恢复了多少条历史记录、哪些 ID），而非仅写一条"已回放"
4. **导出留痕**：每次导出操作生成 `export` 类型日志，包含导出时的完整上下文
5. **版本不兼容提示**：包结构新增 `schemaVersion` 字段（当前版本 `"1.0.0"`），主版本号不匹配时给出明确错误提示并记录 `version_incompatible` 日志，不会静默吞掉
6. **字段缺失提示**：`validatePackageStructure` 对每个缺失的必填字段给出中文名称提示（如"缺少必填字段: 包名称 (name)"），对 `operationLogs` 内每条记录也做字段校验

### 去重规则

| 场景 | 去重策略 |
|------|---------|
| 同一包多次导入 | 按日志 ID 去重，包内已有的历史记录不会重复写入本地；每次导入生成新的 `import` 日志 |
| 同一包多次回放 | 按日志 ID 去重，包内历史记录不会重复；每次回放生成新的 `audit_restore` 日志 |
| 跨机器恢复 | 包内 `operationLogs` 携带全量历史，导入后按 ID 去重合并到本地，时间排序 |

### 新增动作类型

| 动作类型 | 说明 | 触发时机 |
|---------|------|---------|
| `audit_restore` | 审计恢复 | 从会话包回放时，先恢复包内历史记录再写入此日志 |
| `export` | 导出 | 导出会话包为文件时记录 |
| `version_incompatible` | 版本不兼容 | 导入的包 schemaVersion 主版本号与当前应用不匹配时记录 |

### 审计恢复日志上下文

`audit_restore` 日志的 `context` 包含以下额外字段：

```json
{
  "restoredLogCount": 5,
  "restoredLogIds": ["log-1", "log-2", "log-3", "log-4", "log-5"]
}
```

- `restoredLogCount`：从包内恢复的历史记录条数
- `restoredLogIds`：恢复的历史记录 ID 列表

### 版本不兼容日志上下文

`version_incompatible` 日志的 `context` 包含以下额外字段：

```json
{
  "sourcePackageSchemaVersion": "99.0.0",
  "currentSchemaVersion": "1.0.0",
  "incompatibilityReason": "包版本不兼容：包 schema 版本为 99.0.0，当前应用支持 1.0.0（主版本号不匹配：99 vs 1），无法导入"
}
```

会话包 `ReviewSessionPackage` 新增 `schemaVersion` 字段：

```json

  "name": "测试包",
  "version": "1.0.0",
  "schemaVersion": "1.0.0",
  "operationLogs": [ ... ],
  "checksum": "xxx"

```

- `schemaVersion`：包结构版本号，用于兼容性检查

### 冲突处理完整链路（三步独立日志）

1. **冲突检测**（`import_conflict_detected`）：发现版本冲突时立即记录，包含冲突的现有版本号
2. **用户选择**（`import_conflict_cancel` / `import_conflict_rename` / `import_conflict_overwrite`）：用户做出选择后记录，包含选择的解决方式和新版本号（改名时）
3. **最终结果**（`import` / `import_failure`）：导入完成后记录最终结果

## 常见注意事项

**首屏白屏（已修复）**：
- 问题根因：ExportPanel 中通过 `useStore((s) => s.getCurrentJobSnapshots())` 调用方法的方式订阅状态，方法在空状态时每次返回新的 `[]` 引用，配合 Scene3D 每 200ms 一次的相机状态同步，触发无限重渲染
- 修复方案：将所有方法调用式 selector 改为直接订阅基础状态字段 + `useMemo` 派生；Scene3D 中 setCamera 增加浅比较去重，未变化时不触发 store 更新
- 回归测试：`src/__tests__/import-export.test.ts` 中 `regression - infinite re-render / white screen` 分组

**跨作业批注串档 + 失败作业抬头消失（已修复）**：
- 问题根因：① `importJob` 失败分支错误地清空了 `annotations`、`ignoredRiskIds`、`snapshotHistory`，导致提示"当前作业保持不变"但实际上作业内批注、撤销历史全部被清空；② ImportPanel 作业信息区多写了 `errors.length === 0` 条件，一出错就连作业名称、日期、起重机编号一起隐藏
- 修复方案：预检**失败**分支只写 `errors` 和 `lastImportFailure`，当前作业、批注、忽略、快照选中、撤销历史全部不动；作业信息外层显示条件改为仅 `job` 存在就显示，把"数据加载成功"绿勾提示单独包 `errors.length === 0`；只有**成功**导入新作业时才清空 annotations、ignoredRiskIds、snapshotHistory 并重置 currentSnapshotId
- 回归测试：`src/__tests__/import-export.test.ts` 中 `regression - doc-vs-implementation consistency` 分组的 `ONLY on success`、`preserves currentSnapshotId, snapshots bucket and export-ability`、`preserves job.meta so name/date/craneId remain visible` 三条用例
